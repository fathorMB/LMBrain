Overview

LMBrain treats a project's .lmbrain directory as the single source of truth. Every spec, task, decision and review is a Markdown file under version control. This first desktop release is a fast, read-only lens over those files: it never holds hidden state and never writes to the repository.

This keeps the project legible to both humans and coding agents. See [[ADR-001]] for the rationale and [[ADR-002]] for the local-first stance.

Local-first data flow

A file watcher observes the workspace and recomputes derived views (Pulse metrics, backlinks, the taskboard) on change. There is no server and no sync daemon.

# derived state is never persisted
watch('.lmbrain/**/*.md') ─▶ parse ─▶ index ─▶ render
lightbulb
Agents are started manually. The Project Lead prepares specs and reviews completed work; specialists implement from a ready spec. LMBrain surfaces handoffs but never launches an agent on its own — see [[ADR-006]].

The .lmbrain layout

Files live under predictable folders so both the app and agents can navigate by convention:

.lmbrain/
├─ STATUS.md · ROADMAP.md
├─ specs/ready/SPEC-012.md
├─ tasks/in-progress/TASK-031.md
├─ handoffs/active/HANDOFF-001.md
└─ decisions/ agents/ reviews/ mcp/

Scope

Surface derived activity signals for review outcomes and handoff state changes. In this read-only release, signals are computed from existing .lmbrain/ files and displayed in the topbar; no Markdown is created or modified.

Technical proposal

  • Derive activity signals from existing spec, review, task, and handoff metadata.
  • Refresh the in-memory view when the file watcher observes a transition.
  • Render a non-persistent activity indicator in the topbar.
